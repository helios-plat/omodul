## 1. Context

### 1.1 Why Layer 3 Meta-Functions

oprim (Layer 1) 提供 31 个 atomic ops——单个数学操作。
oskill (Layer 2) 提供 13 个工作流 skill——通用组合。

但金融分析中有一组**端到端业务功能**，需要组合多个 skills + ops 才能完成：

- 完整 backtest 报告（CPCV + WFO + per-regime + factor attribution）
- Trade journal 行为偏差诊断（4 类 bias）
- Regime replay 检索 + forward distribution
- Strategy decay 监控
- Scenario stress test（historical + custom + analogy）
- Tail risk 多方法分析
- Calibration engine + Bandit feedback
- ...

如果每个调用方项目自己组合 skills + ops 实现这些功能，会出现：
- **业务功能重复实现**（多项目实现相同 backtest report）
- **集成 bug 重复**（多项目踩同样的坑）
- **API 不一致**（同一功能各项目 schema 不同）
- **真实数据验证缺失**（业务复杂度没有 dogfood 验证）

**omodul 的定位**：通用金融分析元模块库，独立于任何业务项目，任何项目都可调用。

### 1.2 与 oprim / oskill 的关系

```
                调用方业务模块
                  ↓ 自由调用
  ┌───────────────────────────────┐
  │  omodul (Layer 3)     │ ← 本 ADR
  │  16 个端到端业务功能           │
  └─────────────┬─────────────────┘
                ↓ 调用
  ┌───────────────────────────────┐
  │  oskill (Layer 2)       │ ← ADR-062
  │  13 个组合工作流               │
  └─────────────┬─────────────────┘
                ↓ 仅调用
  ┌───────────────────────────────┐
  │  oprim (Layer 1)          │ ← ADR-061
  │  31 atomic ops                 │
  └─────────────┬─────────────────┘
                ↓
  ┌───────────────────────────────┐
  │  Layer 0 标准库                 │
  └───────────────────────────────┘
```

**omodul 设计原则**:
- omodul 是 oskill + oprim 的**最严格 dogfood**
- 业务复杂度集中点（区别于 Layer 1/2 的纯算法层）
- 任何金融分析项目（Helios / Helixa / Tide / Selene / 其它）都可同时 pip install 三个 package

### 1.3 严格定义：什么是 Layer 3 Meta-Function

```yaml
layer_3_definition:

  inclusion (必须全部满足):
    ✅ 端到端可用业务功能（清晰输入 → 可用输出）
    ✅ 跨 ≥ 2 项目可复用（中性，不绑特定项目）
    ✅ 仅依赖 Layer 1 (oprim) + Layer 2 (oskill) + 标准库
    ✅ 不调用其它 Layer 3 element
    ✅ 输入用通用 schema (DataFrame / dict, 不绑特定项目存储)
    ✅ 输出用通用结果 (dict / DataFrame, 不绑特定项目展示)

  exclusion (任一即排除):
    ❌ 单项目专用 → 业务模块层
    ❌ 必须依赖项目特定 schema → 抽象不到位
    ❌ 输出绑特定 UI 格式 → 业务模块
    ❌ 调用其它 Layer 3 → 退化（违反纪律）
    ❌ 业务概念绑定（e.g. Helios "Fusion Score"）→ 业务模块

  positive_examples:
    omodul.trade_journal_analyzer
    omodul.regime_replay_search
    omodul.strategy_backtest_report
    
  excluded_as_business_module:
    helios_fusion_decision_board     # 绑 Helios UI 业务
    helixa_strategy_state_machine    # 绑 Helixa 4-state lifecycle
    tide_a_share_panel               # 绑 Tide A 股 UI

  ⚠️ uncertain_boundary_cases:
    standardized_performance_report:
      可能违反"不调其它 Layer 3" → 重新设计后入选 v0.1
      v0.1 → 1.0 升级前必须验证设计合规性
    tradingview_signal_export:
      几乎不调 oskill/ops → 可能更适合 helios-tools
      v0.1 → 1.0 升级前必须决定归属
```

## 2. Decision

### 2.1 创建独立 `omodul` package, 包含 16 个 Layer 3 元模块

```
仓库: github.com/helios-plat/omodul (独立仓库)
包名: omodul (PyPI 风格命名)
import 名: omodul (Python 模块名)
发布渠道: GitHub Packages
版本: 0.1.0 (候选实施版 → 1.0 需验证)
license: TBD (与 oprim/skills 一致)
依赖: 
  oprim>=1.0.0,<2.0.0
  oskill>=1.0.0,<2.0.0
```

### 2.2 三大架构纪律（继承 + 强化）

```yaml
discipline_1_layer_3_no_internal_calls:
  rule: omodul 内部 element 之间不允许互相 import
  enforcement:
    - PR lint check: 任何 omodul/*.py 文件 import omodul.* → fail
    - CI 阻断 merge

discipline_2_must_use_lower_layers:
  rule: omodul 内部必须 import oskill.* 和/或 oprim.* 实现核心
  enforcement:
    - PR lint warning: omodul/*.py 直接 import scipy.stats.bootstrap 等被 oprim 包装的功能 → warning
    - PR review 强制
  rationale:
    omodul 价值在于"严格基于 Layer 1 + Layer 2"
    一旦绕过下层, 出现两套 bootstrap 实现, 一致性丢失

discipline_3_real_world_dogfood:
  rule: 每个 feature 必须有真实金融数据测试 (不能只用 mock)
  enforcement:
    - tests/ 中必须包含 real_data fixtures (e.g. SPY 252 日 returns)
    - 集成测试不允许 mock oskill / oprim
    - 至少 1 个端到端真实数据 happy path 用例
  rationale:
    Layer 3 业务复杂度高
    单元测试 + mock 不足以验证业务正确性
    必须真实数据端到端验证

discipline_4_neutral_design:
  rule: API 不假设调用方业务上下文
  enforcement:
    - 输入用通用 schema (DataFrame / dict, 含 schema 文档)
    - 输出用通用结果 (dict / DataFrame, 不绑特定 UI)
    - 文档示例用通用金融场景, 不绑特定项目
```

### 2.3 库的独立性原则（继承 oprim / skills）

```yaml
independence_principles:

  no_business_assumption:
    omodul 不假设调用方业务上下文
    API 设计基于"金融分析通用业务功能", 不基于特定项目需求

  no_runtime_dependency:
    不读环境变量 / 配置文件 / 数据库 / 网络
    pure Python functions only

  no_business_logic:
    每个 feature 是组合工作流, 不含业务专属概念
    业务逻辑由调用方组合 features + skills + ops 构造

  no_layer_coupling:
    omodul release 节奏与 oprim/skills 解耦
    oprim/skills 1.x → 2.0 时, omodul 决定何时 migrate

  versioning_independence:
    omodul 0.1 / 0.2 / 1.0 / 1.1 / 2.0 ... 独立演化
    semantic versioning
    pin Layer 1+2 major version
```

### 组 1：交易行为分析（Trade Behavior Analysis）—— 2 个

#### 3.1 `features.trade_journal_analyzer`

**用途**：交易记录 → 4 类常见行为偏差诊断 + 行为模式分析

**调用 oskill + ops**:
- `oskill.detect_outliers_robust`（异常交易检测）
- `oskill.distribution_shift_test`（行为模式漂移）
- `oskill.bootstrap_distribution`（统计量 CI）
- `oprim.percentile_rank`、`oprim.zscore_normalize`

**4 类 Bias 学术定义**:
```
1. Disposition Effect (Odean 1998):
   PGR = 已实现盈利交易数 / (已实现盈利 + 未实现盈利)
   PLR = 已实现亏损交易数 / (已实现亏损 + 未实现亏损)
   DE_score = PGR - PLR (>0 处置效应强)

2. Overtrading:
   turnover_ratio = total_volume / avg_portfolio_value
   z-score vs historical baseline

3. Chasing Momentum:
   momentum_correlation = corr(prior_N_day_momentum, trade_direction)

4. Anchoring:
   exit_price 集中在 entry_price ± 5% 的比例
```

**API**:
```python
def trade_journal_analyzer(
    trades: pd.DataFrame,  # required cols: timestamp, symbol, side, quantity, price
    *,
    diagnostics: list = ["disposition", "overtrading", "chasing", "anchoring"],
    benchmark_returns: pd.Series | None = None,
    lookback_momentum: int = 5,
    bootstrap_ci: bool = True,
    n_bootstrap: int = 1000,
    random_state: int | None = None,
) -> dict
```

**返回**:
```python
{
    "diagnostics": {
        "disposition": {pgr, plr, de_score, ci_low, ci_high, interpretation, n_trades_used},
        "overtrading": {turnover_ratio, zscore_vs_history, interpretation, n_periods},
        "chasing": {momentum_correlation, ci_low, ci_high, interpretation},
        "anchoring": {exit_price_concentration, anchor_zones_identified, interpretation},
    },
    "behavior_metrics": {n_trades_total, avg_holding_days, win_rate, profit_factor, ...},
    "summary_report": {primary_biases, n_trades_analyzed, analysis_period, warnings},
}
```

**学术参考**: Odean 1998《Are Investors Reluctant to Realize Their Losses?》

**跨项目用例**:
- Helios（如有"我的交易"功能）
- Helixa（自我诊断 paper trading 行为）
- Tide（A 股版同样功能）
- 任何含交易记录的项目

#### 3.2 `features.shadow_account_simulator`

**用途**：基于规则的"影子账户"模拟（"如果你严格按规则交易会怎样"）

**调用 oskill + ops**:
- `oskill.regime_aware_performance`
- `oskill.bootstrap_distribution`
- `oprim.cumulative_returns`、`oprim.drawdown_curve`

**核心逻辑**:
```
输入: actual_trades + market_data + rule_fn
对每个时间点:
  shadow_decision = rule_fn(timestamp, context)
  actual_decision = actual_trades 中对应交易
  比较 → rule_violations + pnl_difference
聚合: actual_equity vs shadow_equity 全程对比
```

**API**:
```python
def shadow_account_simulator(
    actual_trades: pd.DataFrame,
    market_data: pd.DataFrame,
    rule_fn: Callable[[pd.Timestamp, dict], dict | None],
    *,
    initial_capital: float = 100000.0,
    regime_labels: pd.Series | None = None,
    bootstrap_significance: bool = True,
    n_bootstrap: int = 1000,
) -> dict
```

**返回**:
```python
{
    "actual_performance": {total_return, sharpe, max_drawdown, n_trades},
    "shadow_performance": {total_return, sharpe, max_drawdown, n_trades},
    "comparison": {pnl_difference, pnl_difference_ci, rule_violations, ...},
    "regime_breakdown": pd.DataFrame | None,
    "actual_equity_curve": pd.Series,
    "shadow_equity_curve": pd.Series,
}
```

**跨项目用例**:
- Helios（教育用）
- Helixa（评估策略偏离）
- 任何含规则系统的项目

#### 3.3 `features.regime_replay_search`

**用途**：当前 panel → 历史最相似时期 + 后续演化分布

**调用 oskill + ops**:
- `oskill.historical_analogy_search`（核心）
- `oskill.regime_transition_analysis`
- `oskill.bootstrap_distribution`

```python
    methods: list = ["dtw", "wasserstein"],
    ensemble: Literal["mean_rank", "borda", "weighted"] = "mean_rank",

) -> dict
```

**返回**:
```python
{
    "top_k_matches": list[dict],
    "forward_distribution": pd.DataFrame,  # day × q_05/25/50/75/95/mean/std
    "cumulative_forward": {expected_return_at_horizon, ci_low, ci_high, probability_positive},
    "regime_transition_summary": dict | None,
    "n_matches_used": int,
}
```

**跨项目用例**:

#### 3.4 `features.regime_change_detector`

**用途**：实时检测 regime 切换 + 切换前后差异分析

**调用 oskill + ops**:
- `oskill.distribution_shift_test`
- `oskill.regime_transition_analysis`
- `oprim.regime_filter_data`、`oprim.regime_label_align`

**返回**:
```python
{

    "n_transitions": int,
    "transition_history_summary": dict | None,
}
```

**跨项目用例**:

#### 3.7 `features.strategy_decay_monitor`

**用途**：策略衰退监控（rolling Sharpe + Mann-Kendall trend + distribution shift + 4 状态机）

**调用 oskill + ops**:
- `oskill.bootstrap_sharpe`（rolling apply）
- `oskill.distribution_shift_test`
- `oprim.mann_kendall_trend`
- `oprim.zscore_normalize`

**4 状态机**:
```
HEALTHY:    trend stable + distribution match + Sharpe stable
DEGRADING:  trend decreasing OR distribution shift detected
CRITICAL:   trend strongly decreasing AND distribution shift
DEAD:       rolling Sharpe < threshold for N consecutive periods
```

**API**:
```python
def strategy_decay_monitor(
    live_returns: pd.Series,
    baseline_returns: pd.Series,
    *,
    rolling_window: int = 60,
    sharpe_threshold_dead: float = 0.0,
    consecutive_periods_dead: int = 30,
    annualization_factor: float = 252.0,
    mk_alpha: float = 0.05,
    shift_alpha: float = 0.05,
) -> dict
```

**返回**:
```python
{
    "decay_state": Literal["HEALTHY", "DEGRADING", "CRITICAL", "DEAD"],
    "decay_score": float,   # 0=healthy, 1=dead
    "rolling_sharpe": pd.Series,
    "rolling_sharpe_ci_low": pd.Series,
    "rolling_sharpe_ci_high": pd.Series,
    "trend_test": dict,
    "distribution_shift_test": dict,
    "consecutive_below_threshold": int,
    "diagnostics": {trend_significant, trend_direction, shift_detected, below_threshold_now},
    "alert_message": str,
}
```

**跨项目用例**:
- Helios §9.8 Strategy Decay
- Helixa 策略 lifecycle 监控
- Selene

#### 3.8 `features.factor_attribution_report`

**用途**：完整因子归因报告（多 factor models 对比 + bootstrap CI + 残差分析）

**调用 oskill + ops**:
- `oskill.factor_attribution`（多次调用不同 factor sets）
- `oskill.distribution_shift_test`（残差检验）
- `oprim.mann_kendall_trend`（rolling alpha trend）
- `oprim.kolmogorov_smirnov_test`（残差正态性）

**Model Comparison 逻辑**:
```
最优 R²:        max(adj_r_squared)
最稳健 alpha:   alpha CI 不含 0 + tstat 最高
最简洁:         factors 数量 vs R² 帕累托最优
```

**API**:
```python
def factor_attribution_report(
    asset_returns: pd.Series,
    factor_sets: dict[str, pd.DataFrame],  # e.g. {"FF3": ..., "FF5": ..., "Carhart4": ...}
    *,
    bootstrap_ci_enabled: bool = True,
    n_bootstrap: int = 1000,
    include_residual_analysis: bool = True,
    include_rolling_alpha: bool = True,
    rolling_window: int = 60,
    standard_errors: Literal["ols", "white", "newey_west"] = "newey_west",
) -> dict
```

**返回**:
```python
{
    "models": dict[str, dict],  # 每个: {factor_attribution_output, residual_analysis, rolling_alpha}
    "model_comparison": {best_r_squared, most_robust_alpha, most_parsimonious},
    "summary": dict,
    "warnings": list[str],
}
```

## 8. Alternatives Considered

### 8.1 Alternative A: 不抽 Layer 3, 业务模块直接组合 Layer 1+2

```yaml
pros:
  - 更直接, 无中间层
  - 业务模块完全自由

  - 多个项目重复实现相同业务功能 (4 项目 × 完整 backtest report)
  - 测试覆盖率参差

verdict: 拒绝 (违反三层栈范式)
```

### 8.2 Alternative B: Layer 3 与 Layer 2 合并

```yaml
pros:
  - 单一组合层

cons:
  - 颗粒度混乱: skills (cpcv_pipeline) vs features (strategy_backtest_report)
  - 设计哲学不同: skills 是工作流, features 是端到端业务功能
  - 真实数据 dogfood 要求不同: skills 用 mock, features 不用

verdict: 拒绝 (颗粒度不同)
```

### 8.3 Alternative C: 等 evaluation extraction 完成再写 ADR-063

```yaml
pros:
  - 基于真实业务代码, 设计更准

  - 与 4 项目暂停战略冲突 (要快速建栈)
  - 0.1 候选实施版本来就是预期需要 dogfood 验证的

  采用方案: 0.1 候选实施 → dogfood → 1.0 正式
```

### 8.4 Alternative D: 只 ship 14 个非待评估 features

## END OF ADR-063

**核心一句话**: omodul 是独立 standalone library，包含 16 个端到端金融分析元模块（组合 oprim + oskill），任何项目都可 `pip install omodul` 后调用。v0.1 候选实施版需经过 dogfood 验证后升级 1.0。

**三层栈完整状态**:
- Layer 1 oprim: ADR-061 + 31 ops + ship-ready
- Layer 2 oskill: ADR-062 + 13 skills + ship-ready
- Layer 3 omodul: ADR-063 (本文档) + 16 features + v0.1 候选实施版

**完整三层栈 ship 后**: 60 个可调用 elements，任何金融分析项目获得完整工程基础设施。
